Generated by [yolo_finetune/scripts/ocr/build_pilot_review_html.py](../../yolo_finetune/scripts/ocr/build_pilot_review_html.py). Regenerate any time with:

**UI features:**

- Fixed 3-column grid, cards up to 1800px wide total
- **Auto-approve on scroll-past**: IntersectionObserver marks a card `approved=true, auto_approved=true` once it fully exits the top of the viewport, unless already flagged
- Auto-approved cards: dashed green outline + 0.55 opacity so they're visually distinct from manually approved (solid green)
- **Per-card controls** (four buttons side-by-side):
  - `✓ Approve` — manual approval (solid green outline)
  - `↻ Rotation` — flag for rotation re-pass (blue outline)
  - `⟲ Opus` — flag for Opus stage-2 review (red outline)
  - `↶ Reset` — clear all state for this card
  - Flags are mutually exclusive — setting any one clears the others
- **Inline-editable OCR text**: `contenteditable="plaintext-only"`. Click to edit, Enter/blur to save, Escape to revert. Edited cards get a cyan text border. Edits persist even after approval.
- **Rotation override**: per-card 0/90/180/270 buttons below the control row. Writes `rotation_override` to the state record. Reverting to the CNN value clears the override.
- **Status bar** (sticky top): live counts for total / unreviewed / approved / auto / rotation-flagged / opus-flagged / edited
- **Filters**: All / Unreviewed / Flagged / Edited (body-class CSS, no DOM removal)
- **Download** button exports everything to `review_state.json`:

```json
{
  "d1_00000133": { "approved": true, "auto_approved": true },
  "d1_00000456": { "opus_flag": true, "edited_text": "9 6'95" },
  "d1_00000789": { "rotation_flag": true, "rotation_override": 90 }
}
```

Only touched stems are stored — untouched cards don't bloat the file.

**localStorage key:** `pilot_review_state_v2`.

## Next Steps

### Immediate — manual review

1. **Verify HTTP server is live** and the HTML loads at `http://192.168.0.2:8000/output/pilot_review.html`. Restart if needed (command above).
2. **User does the manual review pass** in the browser:
   - Scroll through the 3x3 grid. Unreviewed cards auto-approve as they fall off the top.
   - For any card where the OCR is wrong and you can see the correct date: click the text, type the fix, press Enter.
   - For anything you want Opus to double-check: click `⟲ Opus`.
   - For rotation errors the CNN missed: click a `0/90/180/270` rotation override, and/or click `↻ Rotation` to flag for a rotation re-pass.
   - Use the `Unreviewed` filter to focus on remaining work once you've done some scrolling.
3. **Download** `review_state.json` when the unreviewed count is zero (or when satisfied). Save to `yolo_finetune/state/review_state.json`.

### Then — wire manual state into the orchestrator and run Opus stage-2

4. **Merge manual edits into `ocr_results.json`** (new small script or manual Python):
   - For each stem with `edited_text`, overwrite `results[stem]["text"]` with the edited value
   - Tag those entries with `"source": "manual_edit"` or similar so they survive a re-run
5. **Update the trigger set** for stage-2:
   - Existing `select_review_stems()` auto-triggers (regex/?/conf) ✓ already works
   - **Add** manual flags from `review_state.json` where `opus_flag: true`
   - **Add** rotation-disagreement stems where `rotation_flag: true` OR `rotation_override` != `cnn_rot`
   - This logic goes in `scripts/ocr/orchestrate_ocr.py` (probably a new command `merge-review-state` that reads `review_state.json` and updates flags in memory before `crop-stage2`)
6. **Update `cmd_crop_stage2`** to rotate the source image before cropping when `final_rotation != 0`. Currently it crops without rotating, which would produce upside-down views for Opus.
   - Final rotation = `review_state[stem].rotation_override` if set, else `rotation_predictions[stem].rotation`
   - Rotate with PIL: `img = img.rotate(-rot, expand=True)` (CSS rotate is clockwise positive; PIL rotate is counterclockwise positive, hence the negative)
   - Remember to recompute the bbox after rotation (90°/270° swap x/y; 180° mirrors both)
7. **Run `crop-stage2`** to produce the two-view crops + manifests for flagged stems.
8. **Dispatch Opus stage-2 workers** via the Agent tool with `model: "opus"`, `run_in_background: true`, one per shard. Use [scripts/ocr/subagent_prompts/stage2_prompt.md](../../yolo_finetune/scripts/ocr/subagent_prompts/stage2_prompt.md) as the prompt body.

10. **Final HTML** regenerated with stage-2 verdicts baked in (show `review_status` per card).

## Key Context

### Architecture

- **Sonnet for stage-1, Opus for stage-2** (cost-tier on subscription). Haiku was tried first and misread many `'95`s as `'99`s; Sonnet was consistently correct.
- **OCR is rotation-agnostic**: Sonnet reads rotated 7-segment digits correctly, so we don't need to re-OCR rotated photos. Rotation matters only for display in the review HTML and for the final archive.
- **Three valid stamp formats** discovered this session, all accepted by `DATE_FORMAT_RE`:
  - `M D 'YY` — e.g., `10 3 '99` (space before apostrophe)
  - `M D'YY` — e.g., `9 6'95` (no space)
  - `'YY M D` — e.g., `'94 8 23` (year first, older cameras)
- **Stage-1 trigger rules** (in `should_review()`): contains `?`, or text is neither `NONE` nor matches `DATE_FORMAT_RE`, or YOLO confidence < 0.3
- **Sharding**: 50 stems/stage-1 shard, 25 stems/stage-2 shard. 126 stage-1 shards ran in 12 waves of ~10 parallel Sonnet subagents each.
- **Catch-up merge pattern** — use after any wave:
  ```
  for f in state/shards/stage1/shard_*_result.json; do
    case "$f" in *_sonnet_*|*_gemma4_*|*_qwen3-vl_*) continue ;; esac
    uv run scripts/ocr/orchestrate_ocr.py merge-stage1 "$f"
  done
  ```

### Local VLM benchmark outcome

Ran `gemma4:e4b` and `qwen3-vl:4b` on shard 0000 (50 crops) for comparison with Sonnet.

- **Sonnet** — gold standard
- **Gemma 4 e4b** — 74% match with Sonnet (mostly formatting noise: stray dots, split digits). 18 s/img on CPU → ~32 hours for the full 6,458. Viable local fallback but noisier.
- **Qwen3-VL 4b** — 32% match. Reasoning-token starvation produced 12 empty strings out of 50. 68 s/img. Not worth pursuing.

**Decision:** Sonnet for production. Local VLMs are not competitive enough on this task.

### Gotchas

- **`BASE_DIR` bug fixed in the rewrite**: the old script had `Path(__file__).parent.parent` which worked when it lived at `scripts/build_pilot_review_html.py` but pointed at `scripts/` after the refactor moved it to `scripts/ocr/`. The new version uses `.parent.parent.parent.resolve()`.
- **Pre-existing unrelated modified files** that were NOT touched by this work:
  - (These have been merged/cleaned up since the first handoff since the feature branch was merged. Only `scripts/infer/infer_all.py` still shows as modified — pre-existing unrelated change.)
- **Rotation confidence is conservative** — no predictions ≥ 0.95. Adjust auto-apply threshold to ≥ 0.85 if automating rotation correction downstream.
- **HTML file is 11 MB** because it inlines every card block. At this size it may be slow to parse on weaker clients; consider pagination (e.g., chunked ranges via `?from=0&to=500`) if performance is poor in practice.

### Subagent dispatch pattern (what we did for stage-1 and will repeat for stage-2)

```
Agent(
  description="Sonnet shard NNNN",
  subagent_type="general-purpose",
  model="sonnet",  # or "opus" for stage-2
  run_in_background=True,
  prompt="<stage1_prompt.md contents> + SHARD_MANIFEST_PATH + BASE_DIR"
)
```

Dispatch in waves of 10 per message (the harness queues internally). Merge each shard as its completion notification arrives; batch-catch-up via the shell loop above any time.
